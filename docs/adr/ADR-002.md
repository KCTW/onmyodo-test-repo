# ADR-002: 測試策略——單元測試框架選型與 CI 整合

> **狀態（Status）**: Proposed
> **日期（Date）**: 2026-02-27
> **作者（Author）**: Onmyodo Architect Agent

---

## 背景 (Context)

`onmyodo-test-repo` 是 Onmyodo Sprint 01 所建立的 POC 驗證 repository，
用途為確認 GitHub MCP Server 的讀寫能力（詳見 `README.md`）。

目前 repository 根目錄包含三個不同語言的平行實作：

| 檔案 | 語言 | 核心函式 |
|------|------|----------|
| `sample.py` | Python 3 | `def greet(name: str) -> str` |
| `sample.ts` | TypeScript | `function greet(name: string): string` |
| `sample.go` | Go | `func greet(name string) string` |

經過分析，發現以下問題：

1. **缺乏測試檔案**：三個語言的 sample 檔案皆未附帶對應的測試檔（無 `test_sample.py`、`sample.test.ts`、`sample_test.go`）。
2. **缺乏 CI 設定**：repository 中不存在 `.github/workflows/` 目錄，表示目前沒有任何自動化測試流程。
3. **缺乏套件管理設定**：無 `package.json`（TypeScript）、`go.mod`（Go）或 `requirements.txt`（Python），導致無法重現環境。
4. **純函式結構適合單元測試**：三個 `greet` 函式均為無副作用的純函式（pure function），輸入固定 → 輸出固定，是單元測試的最佳候選。

根據 Onmyodo 專案記憶，**TypeScript 已被確認為主要語言**，但此 repository 屬於多語言驗證場景，三種語言均需納入測試策略考量。

因此，需要為此 polyglot codebase 制定一套統一的測試框架選型原則，並規劃對應的 CI 整合方案。

---

## 決策 (Decision)

### 選定方案

針對三種語言，分別採用各自生態系的主流測試框架，並以 GitHub Actions 作為 CI 平台：

#### TypeScript → **Vitest**

- **選擇理由**：TypeScript 原生支援（無需 `ts-jest` 轉譯設定）；ES Modules 原生相容；API 與 Jest 幾乎一致（學習成本低）；執行速度快（HMR-based）。
- **安裝方式**：`npm install -D vitest`
- **測試指令**：`npx vitest run`
- **對應檔案**：`sample.test.ts`（待建立）

#### Python → **pytest**

- **選擇理由**：Python 生態事實標準；支援型別標註（`sample.py` 已使用 `str` 型別提示）；fixture 機制強大；最小設定即可執行。
- **安裝方式**：`pip install pytest`
- **測試指令**：`pytest`
- **對應檔案**：`test_sample.py`（待建立）

#### Go → **testing（內建標準函式庫）**

- **選擇理由**：Go 官方內建，無需外部依賴；`go test ./...` 一行指令涵蓋所有測試；Table-driven test 模式與 `sample.go` 的純函式結構完美契合。
- **測試指令**：`go test ./...`
- **對應檔案**：`sample_test.go`（待建立）

#### CI 平台 → **GitHub Actions**

- **選擇理由**：repository 已託管於 GitHub，原生整合免額外帳號；公開 repository 免費分鐘數充足；Matrix strategy 可一個 workflow 並行測試三種語言。
- **觸發條件**：`push` 到任何分支、`pull_request` 建立或更新時觸發。

---

### 考慮過的替代方案

| 方案 | 評估結果 | 否決原因 |
|------|----------|----------|
| TypeScript: **Jest** | ❌ | 需要 `ts-jest` 或 Babel 轉譯設定，啟動較慢，設定複雜 |
| TypeScript: **Mocha + Chai** | ❌ | 需多個套件組合，TypeScript 支援需額外設定 |
| Python: **unittest** | ❌ | 內建但語法冗長，缺乏 fixture 支援，不如 pytest 易讀 |
| Python: **nose2** | ❌ | 已進入維護模式，社群活躍度低 |
| CI: **CircleCI** | ❌ | 需建立額外帳號，與 GitHub 整合需要 OAuth，增加維運複雜度 |
| CI: **Travis CI** | ❌ | 開源免費方案已縮減，且整合體驗不如 GitHub Actions 原生 |

---

## 後果 (Consequences)

### ✅ 正面影響

1. **各語言慣用框架**：每種語言均採用其生態系最主流的測試工具，團隊成員無需學習陌生框架。
2. **CI 自動驗證**：每次 PR 均自動執行三語言測試，確保 `sample.py`、`sample.ts`、`sample.go` 的 `greet` 函式行為一致。
3. **Pure function 高可測性**：`greet(name)` 無 I/O 副作用，可用簡單 assertion 完整覆蓋正常路徑與邊界條件。
4. **GitHub Actions Matrix**：單一 `ci.yml` 即可並行驗證三種語言，節省 CI 設定維護成本。

### ⚠️ 負面影響與風險

1. **三套測試環境維護成本**：分別需要 Node.js runtime、Python runtime、Go toolchain，CI image 選型需謹慎（建議使用 `ubuntu-latest` 內建多語言支援）。
2. **套件管理設定缺失**：目前無 `package.json`、`go.mod`、`requirements.txt`，需補齊才能讓 CI 正確安裝依賴。
3. **測試覆蓋率追蹤**：三種語言的覆蓋率報告格式不同，暫不整合 Codecov 等工具（列為 Sprint 02 後續事項）。
