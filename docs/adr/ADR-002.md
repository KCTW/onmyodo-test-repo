# ADR-002: 測試策略 — 單元測試框架選型與 CI 整合

> **狀態 (Status)**: Proposed
> **日期 (Date)**: 2026-02-27
> **作者 (Author)**: Onmyodo Architect Agent

---

## 背景 (Context)

`onmyodo-test-repo` 目前為 Onmyodo POC 階段所建立的驗證 repository，用途為確認 GitHub MCP Server 的讀寫能力（見 `README.md`）。目前 repository 根目錄下包含三個多語言範例源碼檔案：

- **`sample.py`**（Python）：定義 `greet(name: str) -> str` 純函式，以 f-string 回傳問候字串
- **`sample.ts`**（TypeScript）：定義 `greet(name: string): string` 純函式，以 template literal 回傳問候字串
- **`sample.go`**（Go）：定義 `greet(name string) string` 純函式，以 `fmt.Sprintf` 回傳問候字串

三個函式均為 pure function（明確輸入輸出、無副作用），是理想的單元測試標的。

然而，目前 repository 中**完全沒有測試檔案**，亦**無任何 CI 組態**（無 `.github/workflows/`、無 `package.json`、無 `go.mod`、無 `requirements.txt`）。在 Onmyodo 主線開發採用 **TypeScript（Node.js + tsx + Anthropic Agent SDK）** 的既定背景下，必須明確定義各語言的測試框架選型與 CI 整合方案，以確保程式碼品質閘道能在後續 Sprint 中自動執行，落實 SDD → TDD → Implementation 的開發原則。

---

## 決策 (Decision)

### 主要決策

針對 `onmyodo-test-repo` 中三種語言，分別採用以下測試框架，並以 **GitHub Actions** 作為統一 CI 平台：

| 語言 | 測試框架 | 執行指令 |
|---|---|---|
| TypeScript | **Vitest** | `npx vitest run` |
| Python | **pytest** | `pytest` |
| Go | **內建 `testing` 套件** | `go test ./...` |

---

### TypeScript：選用 Vitest（非 Jest）

Onmyodo 主線採用 TypeScript + ESM + `tsx` 執行環境。在此環境下：

- **Jest** 需要額外的 `ts-jest` 轉譯層，與 ESM 模組系統存在已知相容性問題，設定繁瑣
- **Vitest** 原生支援 ESM 與 TypeScript，無需額外 transform 設定，啟動速度更快（基於 Vite 引擎）
- Vitest 的 API 與 Jest 高度相容（`describe`、`it`、`expect`），學習曲線極低

對應 `sample.ts` 中的 `greet()` 函式，測試檔案 `sample.test.ts` 可直接撰寫：

```typescript
import { describe, it, expect } from 'vitest';
import { greet } from './sample';

describe('greet', () => {
  it('應回傳含有名稱的問候字串', () => {
    expect(greet('Onmyodo')).toBe('Hello from Onmyodo, Onmyodo!');
  });
});
```

> **替代方案考量**：**Jest**（ESM 支援差、設定複雜）、**Mocha + Chai**（需手動組合多工具，生態較舊）

---

### Python：選用 pytest

- `sample.py` 中的 `greet()` 為純函式，直接以 pytest 撰寫 assertion
- pytest 為 Python 生態最廣泛採用的測試框架，支援 fixture、參數化測試、豐富插件生態
- 測試檔案 `test_sample.py` 範例：

```python
from sample import greet

def test_greet_returns_greeting():
    assert greet("Onmyodo") == "Hello from Onmyodo, Onmyodo!"
```

> **替代方案考量**：**unittest**（內建但冗長，fixture 不如 pytest 靈活）

---

### Go：選用內建 `testing` 套件

- Go 標準函式庫已內建 `testing` 套件，`go test ./...` 即可運行所有測試
- `sample.go` 的 `greet()` 純函式無外部依賴，不需要 mock 框架
- 測試檔案 `sample_test.go` 範例：

```go
package main

import "testing"

func TestGreet(t *testing.T) {
    got := greet("Onmyodo")
    want := "Hello from Onmyodo, Onmyodo!"
    if got != want {
        t.Errorf("greet() = %q, want %q", got, want)
    }
}
```

> **替代方案考量**：**testify**（assertion 更易讀，適合後期複雜測試場景）、**Ginkgo**（BDD 風格，對 POC 過度複雜）

---

### CI 整合：GitHub Actions

以 GitHub Actions 作為 CI 平台，原因如下：

1. **原生整合**：Onmyodo 整個開發流程（ADR PR、Branch 管理、Issue 追蹤）均在 GitHub 上進行，Actions 無需額外設定憑證
2. **並行執行**：同一 workflow 中以獨立 job 並行測試三種語言，縮短 CI 等待時間
3. **觸發時機**：設定於 `push` 至任意 branch 及 `pull_request` 開啟時自動執行

建議 CI workflow 結構（`.github/workflows/test.yml`）：

```yaml
name: Test

on: [push, pull_request]

jobs:
  test-typescript:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
      - run: npm ci && npx vitest run

  test-python:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.12'
      - run: pip install pytest && pytest

  test-go:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-go@v5
        with:
          go-version: '1.22'
      - run: go test ./...
```

## 後果 (Consequences)

### 正面影響

- **品質閘道自動化**：每次 PR 均自動執行三語言測試，防止回歸錯誤合入 `master` 主線
- **TypeScript 優先對齊**：Vitest 與 Onmyodo 主線 ESM/tsx 環境完全相容，零額外轉譯層設定，符合 MEMORY.md 語言選型決策
- **低摩擦力**：Go 使用內建工具（零依賴）、Python 使用 pytest（最小依賴）、TypeScript 使用 Vitest（最小設定）
- **SDD → TDD 落地基礎**：此 ADR 為後續針對 `sample.ts`、`sample.py`、`sample.go` 撰寫對應測試檔案提供選型依據，直接支持 Test-First 開發原則（Red Lines #7）

### 負面影響／風險

- **多框架維護成本**：三種語言各有學習曲線與依賴版本更新需求；POC 階段尚可接受，正式產品應以 TypeScript 為主力開發語言
- **CI 初始設定工作量**：需建立 `package.json`（Vitest 依賴）、`requirements.txt`（pytest）、`go.mod`，以及 `.github/workflows/test.yml`，需消耗一定 Sprint Story Point
- **多語言技術債風險**：此 repo 為測試驗證用途，正式 Onmyodo 產品應統一以 TypeScript 為主，避免多語言維護造成技術債積累

### 後續行動

1. 建立 `package.json` 並加入 Vitest 依賴，撰寫 `sample.test.ts`
2. 建立 `requirements.txt` 加入 pytest，撰寫 `test_sample.py`
3. 建立 `go.mod`，撰寫 `sample_test.go`
4. 建立 `.github/workflows/test.yml`，整合三語言並行 CI jobs
5. 於 `scripts/validate_commit.sh` pre-commit hook 加入本地 TypeScript 測試執行（作為 CI 前的本地快速驗證）
