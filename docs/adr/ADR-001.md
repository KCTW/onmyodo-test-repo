# ADR-001: 技術堆疊分析——程式語言多樣性與專案結構

> **狀態（Status）**: Proposed
> **日期（Date）**: 2026-02-27
> **作者（Author）**: Onmyodo Architect Agent

---

## 背景 (Context)

`KCTW/onmyodo-test-repo` 是 Onmyodo Sprint 01 所建立的 POC 測試 Repository，主要目的是驗證 GitHub MCP Server 的讀寫能力（詳見 `README.md`）。其 `README.md` 明確說明：

> 用途：驗證 GitHub MCP Server 的讀寫能力，包括讀取多語言檔案（Python / TypeScript / Go）、建立 Branch 與 Pull Request、Issue 操作。

目前 Repository 根目錄包含以下四個檔案：

| 檔案 | 語言 | 大小 |
|------|------|------|
| `README.md` | Markdown | 299 bytes |
| `sample.py` | Python | 165 bytes |
| `sample.ts` | TypeScript | 155 bytes |
| `sample.go` | Go | 192 bytes |

三個 sample 檔案各自以不同語言實作相同的 `greet(name)` 函式模式：

```python
# sample.py — 使用 f-string 與 PEP 526 型別提示
def greet(name: str) -> str:
    return f"Hello from Onmyodo, {name}!"
```

```typescript
// sample.ts — 使用 ES6 Template Literal 與 TypeScript 靜態型別
function greet(name: string): string {
  return `Hello from Onmyodo, ${name}!`;
}
```

```go
// sample.go — 使用 fmt.Sprintf 與 Go 靜態型別
func greet(name string) string {
    return fmt.Sprintf("Hello from Onmyodo, %s!", name)
}
```

觀察到以下結構問題：

1. **扁平結構**：所有檔案位於根目錄，無任何子目錄（`src/`、`lib/`、`docs/`、`tests/` 均不存在）
2. **無建置工具設定**：缺乏 `package.json`、`go.mod`、`requirements.txt`、`tsconfig.json` 等套件管理或編譯設定
3. **無測試框架**：三個 sample 檔案均無對應的單元測試
4. **三語言並存**：Python、TypeScript、Go 同時出現，但無任何跨語言整合機制

隨著 Onmyodo 從 POC 測試階段邁向正式開發，需要對此 Repository 的語言選型與專案結構作出明確決策，以避免技術債累積與維護成本膨脹。

---

## 決策 (Decision)

**決策：以 TypeScript 作為 onmyodo-test-repo 及 Onmyodo 平台的主要開發語言，並建立標準目錄結構。**

### 考量的替代方案

| 方案 | 描述 | 評估 |
|------|------|------|
| **維持三語言並存** | 保留 `sample.py`、`sample.ts`、`sample.go` 三種語言 | ❌ 維護成本高，技術棧分散，小型 AI 自動化團隊難以跨語言維持品質 |
| **採用 Python** | 以 Python 為主要語言取代其他語言 | ⚠️ AI/ML 生態豐富，但 Anthropic Agent SDK 以 TypeScript/JavaScript 為主要支援目標，長期整合複雜 |
| **採用 Go** | 以 Go 為主要語言取代其他語言 | ⚠️ 執行效能優異，但 AI Agent 生態成熟度不足，Anthropic 無官方 Go SDK |
| **採用 TypeScript** ✅ | 以 TypeScript 為唯一主要語言 | ✅ Anthropic Agent SDK 原生支援、Node.js 生態完整、靜態型別有助大規模 Agent 協作維護 |

### 決策理由

1. **Anthropic Agent SDK 原生支援**：Onmyodo 核心功能依賴 Anthropic Claude Agent SDK，其 TypeScript/JavaScript 版本擁有最完整的官方文件與持續更新
2. **型別安全**：TypeScript 的靜態型別系統（如 `sample.ts` 中的 `name: string` 回傳型別標注）有助在大型 Agent 協作架構中提前捕捉介面錯誤
3. **工具鏈驗證**：GitHub MCP Server、Node.js 工具鏈（`tsx`、`npm`）已於 Sprint 01 環境中完成驗證
4. **團隊一致性**：避免 Scrum Agent 成員跨語言切換所造成的認知負擔與 Code Review 困難
5. **生態系整合**：TypeScript 可直接調用 Node.js 生態（HTTP、檔案系統、CI/CD 工具），並藉由型別定義檔（`.d.ts`）整合現有 JavaScript 套件

### 結構決策

採用以下標準目錄結構，取代目前的扁平根目錄佈局：

```
onmyodo-test-repo/
├── docs/
│   └── adr/              ← Architecture Decision Records（本 ADR 所在位置）
├── src/                  ← TypeScript 原始碼
├── tests/                ← 單元測試（Jest 或 Vitest）
├── scripts/              ← 工具腳本（validate_commit.sh 等）
├── README.md
├── package.json          ← Node.js 套件設定
├── tsconfig.json         ← TypeScript 編譯設定
└── .gitignore            ← 排除 node_modules/、dist/ 等產出物
```

## 後果 (Consequences)

### 正面影響 ✅

1. **降低認知負擔**：全團隊（含 AI Scrum Agents）只需熟悉一種語言與工具鏈，減少跨語言切換
2. **統一 CI/CD 流程**：單一語言簡化 GitHub Actions Workflow 設定，避免多語言 Runner 並行管理
3. **型別安全**：TypeScript 嚴格模式（`strict: true`）可在編譯期捕捉大量介面不匹配錯誤，提升 Agent 間通訊可靠性
4. **ADR 目錄建立**：本 ADR 同時確立 `docs/adr/` 目錄慣例，為後續所有架構決策提供統一存放位置
5. **可測試性**：集中至 TypeScript 後，可引入 Jest/Vitest 建立完整的自動化測試覆蓋率

### 負面影響 ⚠️

1. **遷移成本**：現有 `sample.py` 與 `sample.go` 需評估去留（作為歷史參考保留，或確認 MCP 測試目的完成後移除）
2. **Go 效能優勢放棄**：若未來 Onmyodo 需要高吞吐量微服務元件，重新引入 Go 的決策成本將提高
3. **Python AI 生態受限**：若需直接調用僅支援 Python 的 AI 函式庫（如特定 LangChain 模組），需額外建立跨語言包裝層

| P1 | 評估並決定 `sample.py`、`sample.go` 去留（歸檔或刪除） | Architect Agent |
