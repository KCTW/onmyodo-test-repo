# ADR-001: Tech Stack Analysis — Programming Language Diversity and Project Structure

> **Status**: Proposed
> **Date**: 2026-02-27
> **Author**: Architect Agent (Onmyodo AI Scrum Team)

---

## 背景 (Context)

`onmyodo-test-repo` was automatically created during **Onmyodo Sprint 01** (see `README.md` line 8: *"此 repo 由 Onmyodo Sprint 01 自動建立"*) specifically to validate the GitHub MCP Server's read/write capabilities. As stated in the README, its declared purposes are:

1. Reading multi-language source files (Python / TypeScript / Go)
2. Creating branches and pull requests programmatically
3. Performing Issue operations via the MCP Server API

The repository currently contains **four files, all at the root level**, with no subdirectories:

| File | Language | Size |
|------|----------|------|
| `README.md` | Markdown | 299 bytes |
| `sample.py` | Python | 165 bytes |
| `sample.ts` | TypeScript | 155 bytes |
| `sample.go` | Go | 192 bytes |

### Observed Code Patterns

All three source files implement an **identical logical structure** — a `greet(name)` function returning the string `"Hello from Onmyodo, {name}!"` — expressed in each language's idiomatic style:

- **`sample.py`**: Uses Python type hints (`name: str -> str`) and an f-string; includes `if __name__ == "__main__"` guard.
- **`sample.ts`**: Uses TypeScript type annotations (`name: string: string`) and a template literal; direct `console.log` execution.
- **`sample.go`**: Uses `package main` + `import "fmt"`, `fmt.Sprintf` for interpolation, standard `main()` entry point.

### Structural Gaps Observed

- No **package manager manifests** (`package.json`, `go.mod`, `requirements.txt`, `pyproject.toml`)
- No **build tooling** or compiler configuration
- No **test files** of any kind
- No **CI/CD pipeline** (`.github/workflows/`)
- No **linting or formatting config** (`.eslintrc`, `gofmt`, `ruff`, etc.)
- The `docs/adr/` directory (this file's location) did not previously exist

This absence of tooling confirms the repo's role as a **minimal test fixture**, not a production-grade codebase.

---

## 決策 (Decision)

### Core Decision

**`onmyodo-test-repo` is classified as a read/write test fixture for MCP Server validation only.** The co-existence of Python, TypeScript, and Go source files does not imply polyglot intent for production Onmyodo code. The three languages serve as a coverage matrix to verify the MCP Server's language-agnostic file-reading capability.

**TypeScript remains the confirmed primary language for all Onmyodo production code**, as established in Sprint 01 Task T01-1 (語言選型).

### Alternatives Considered

| Alternative | Assessment | Decision |
|---|---|---|
| **A1**: Treat Python/Go as active production candidates alongside TypeScript | Conflicts with T01-1 decision; introduces toolchain complexity with no benefit | ❌ Rejected |
| **A2**: Collapse all samples into a single polyglot file or script | Defeats the MCP read-coverage test purpose; MCP needs distinct file paths per extension | ❌ Rejected |
| **A3**: Promote flat root structure to production pattern | Not scalable; violates `file_structure.md` layout; no separation of concerns | ❌ Rejected |
| **A4** *(chosen)*: Acknowledge repo as test fixture; maintain language samples as-is; document intent via ADR | Preserves MCP validation utility; avoids confusion; low cost | ✅ **Accepted** |

### Rationale

The parallel structure of the three `greet()` samples (same function, same output, three syntaxes) is a deliberate test matrix, not evidence of architectural pluralism. The complete absence of any build tooling, dependency manifests, or tests further confirms that this repo is not a development target — it is an **instrument** for verifying the tooling layer (GitHub MCP Server) that Onmyodo's AI agents rely upon.

---

## 後果 (Consequences)

### ✅ Positive

- **MCP Server capability confirmed** across three distinct language runtimes (Python, TypeScript, Go), validating Sprint 01 US-01 acceptance criteria.
- **No tech-debt risk**: the simplicity of the fixture means it will not need maintenance beyond its stated test purposes.
- **ADR baseline established**: creating `docs/adr/` sets a documentation pattern that can be adopted in the main Onmyodo production repository.
- **Architectural clarity**: formally documents that TypeScript is the sole production language, preventing future ambiguity from the multi-language presence in this repo.

### ⚠️ Negative / Risks

- **Contributor confusion risk**: developers encountering this repo without context may incorrectly infer that Onmyodo is a polyglot system. The README note ("請勿手動修改") partially mitigates this, but the intent could be more explicit.
- **Non-runnable samples**: without `go.mod`, `package.json`, or `requirements.txt`, none of the three sample files can be executed via standard toolchain commands (`go run`, `ts-node`, `python`... the Python one is the only exception as it has no dependencies).
- **Flat root structure**: all files at root is acceptable for a 4-file fixture but would be unworkable if the test suite grows (e.g., adding JSON, YAML, or binary file tests).

### 🔁 Follow-Up Actions

1. **README update** (Low priority): Add a sentence clarifying that TypeScript is Onmyodo's primary production language; the Python/Go files exist solely for MCP read-coverage testing.
2. **If fixture grows**: introduce a `samples/` subdirectory (e.g., `samples/python/`, `samples/typescript/`, `samples/go/`) to maintain a clean root.
3. **Production repo**: ensure the main `KCTW/onmyodo` repository adopts the directory structure defined in `file_structure.md` and does **not** replicate the flat layout of this test fixture.
4. **CI consideration**: if this repo is used in automated Sprint testing pipelines, add a minimal `.github/workflows/mcp-validation.yml` to surface failures early.
